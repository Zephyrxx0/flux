<threat_model>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| npm registry → local install | Unverified packages cross into build system |
| Zod schema ↔ TypeScript type | Schema mismatch causes silent data corruption at runtime |
| Client bundle ↔ Server-only code | Client delivery of Node.js built-ins causes bundle crash |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-01 | Tampering | Zod schemas in `src/types/` | mitigate | Single barrel export (`src/types/index.ts`) ensures server route handlers and client components use identical schemas. All consumers import from barrel, never from individual files. |
| T-13-02 | Information Disclosure | Route handler ↔ client import boundary | mitigate | New route handler files (`src/app/api/*/route.ts`) are kept self-contained with no shared imports. Client components use `fetch()` to call APIs, never import route handlers. |
| T-13-03 | Elevation of Privilege | Docker build SSL verification | accept | All npm packages are verified via lockfile (`npm ci`). No new packages added in this phase. Risks addressed in research Package Legitimacy Audit. |
| T-13-SC | Tampering | npm packages during build | mitigate | All packages are existing deps with lockfile verification. No `[SLOP]` packages. `[ASSUMED]` packages (next, typescript) are pre-approved per research audit — no blocking checkpoint needed for Phase 13. |

</threat_model>

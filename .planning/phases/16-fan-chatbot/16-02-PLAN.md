<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client→API | POST body with user messages — unvalidated text enters prompt. Validated via ChatRequestSchema (structure, max items). Content is unvalidated (text is freeform). |
| API→Gemini | Full prompt (user message + zone data + match state + history) sent to external Gemini API |
| Gemini→API | Structured JSON response — validated via parseChatResponse() with retry on failure |
| API→client | SSE stream of tokens and structured data — no env vars or internal state leaked |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-16-01 | Information Disclosure | `process.env.GEMINI_API_KEY` in route handler | mitigate | Server-only route.ts (Next.js server boundary). Env var used in fetch URL but never returned to client. Error events say "API key not configured". per D-16-02 |
| T-16-02 | Tampering | `parseChatResponse()` Zod validation | mitigate | ChatResponseSchema safeParse() validates every Gemini response. AI-SPEC §4b retry strategy: 1 retry on failure. Text-only fallback if both attempts fail. |
| T-16-03 | Denial of Service | POST /api/chat (no auth) | accept | No auth on fan route — accept risk for MVP. Rate limiting can be added post-MVP via middleware. |
| T-16-04 | Spoofing | Prompt injection via user message | mitigate | System prompt restricts to zone/match data only. Retry does not relax constraints. AI-SPEC §6 online scope guard |
| T-16-05 | Repudiation | Gemini hallucination | mitigate | ChatResponseSchema validates structure. System prompt prohibits guessing amenities. per AI-SPEC §4 |
</threat_model>

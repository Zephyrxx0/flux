    3. Export `async function GET(_request: NextRequest)`:
       - Read `process.env.WORLDCUP26_TOKEN` — if empty/missing, return `Response.json({ error: "WORLDCUP26_TOKEN not configured", status: "error" }, { status: 500 })`
       - Try block: fetch `https://worldcup26.ir/get/games` with `Authorization: Bearer ${token}` header and `AbortSignal.timeout(10_000)`
       - If `!response.ok`: return `{ error: \`Upstream returned ${response.status}\`, status: "upstream_error" }` with the upstream status code
       - `const raw = await response.json() as unknown`
       - `WorldCup26GamesSchema.safeParse(raw)` — if parse fails, return `{ error: "Invalid upstream data", status: "parse_error", issues: parsed.error.issues }` with status 502
       - `parsed.data.games` → call `findLiveMatch(games)` → map to MatchState with `mapGameToMatchState` if found, else null
       - Build `allGames` summary array mapping each game to `{ homeTeam, awayTeam, localDate, timeElapsed, finished, score }`
       - Return `Response.json({ match: matchState, allGames })`
       - Catch: return `{ error: err.message, status: "fetch_error" }` with status 502

    Error responses follow the consistent `{ error, status }` discriminator pattern documented in PATTERNS.md. Do NOT include the token or full game data in error responses per security rules.

    The response shape is: `{ match: LiveMatch | null, allGames: Array<{ homeTeam: string, awayTeam: string, localDate: string, timeElapsed: string, finished: string, score: string }> }`

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-14-01 | Information Disclosure | route.ts env var access | mitigate | WORLDCUP26_TOKEN read only via process.env in server route handler. `import "server-only"` prevents accidental client import. Token NEVER included in response. |
| T-14-02 | Denial of Service | worldcup26.ir fetch | mitigate | AbortSignal.timeout(10_000) prevents hanging connections. 401 response caught and returned as upstream_error — no retry loop on server. |
| T-14-03 | Information Disclosure | error response | mitigate | Error responses return opaque `"upstream_error"` / `"parse_error"` discriminators. Raw fetch or Zod errors are NOT forwarded to client — only the error message string. Token never appears in any error response. |
| T-14-04 | Tampering | Zod safeParse | mitigate | WorldCup26GamesSchema validated with safeParse before any processing. Malformed upstream data returns 502 parse_error — never forwarded to client. |

    Actually, looking more carefully: the API returns `{ match: LiveMatch | null, allGames: [...] }`. So the fetchFn should succeed (not throw) even when match is null. The hook's callback ref pattern calls fetchFn, and the result would be `MatchState | null`. But MatchState is defined as non-null...

    Let me reconsider. The fetchFn for useMatchPoller should:
    - Call /api/match
    - If the response has match data, return it
    - If match is null (no live match), that's still a valid response — the upcoming "state" would need allGames data too

    For Phase 14 simplicity: the fetchFn fetches /api/match, checks status, parses JSON. The response always succeeds (even with match: null). The useMatchPoller receives the match object or null. MatchBanner reads from the store. The upcoming/error/loading states are determined by the store's match state plus the poller's error state.

    This is a known simplification — the full upcoming state with allGames list will require store expansion. For Phase 14 scope:
    - If match is non-null: live state
    - If match is null and no error: upcoming/empty state (or loading initially)
    - If error: error state

    Let me make the fetchFn return the full response and store relevant data. The cleanest approach:

    ```typescript
    const matchPoller = useMatchPoller(async () => {
      const res = await fetch("/api/match")
      if (!res.ok) throw new Error(`Poll failed: ${res.status}`)
      return res.json()  // { match: LiveMatch | null, allGames: [...] }
    })
    ```

    But then the hook's PollerState would need a different type... This is getting complicated for the strict separation. Let me simplify:

    The useMatchPoller's fetchFn returns `MatchState`. The dashboard page calls fetch with a fetchFn that:
    1. Fetches /api/match
    2. Validates response
    3. Calls `setMatch(json.match)` on the store
    4. Returns json.match (or throws if null)

    This way MatchBanner reads from the store, not from the hook's return value. The hook's data property is secondary.

    Actually even simpler: the dashboard page handles the fetch in a custom wrapper that updates the store:

    ```typescript
    const fetchMatch = useCallback(async () => {
      const res = await fetch("/api/match")
      if (!res.ok) throw new Error(`Poll failed: ${res.status}`)
      const json = await res.json()
      const matchState = json.match
      liveStore.getState().setMatch(matchState)
      return matchState
    }, [])
    
    const pollerState = useMatchPoller(fetchMatch)
    ```

    This is clean. The store updates trigger MatchBanner re-renders automatically.

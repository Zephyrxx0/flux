  <action>
    Create `src/hooks/useWeather.ts` modeled exactly on `useMatchPoller.ts` with these adaptations (D-17-09):

    **Exports:**
    ```typescript
    export interface WeatherPollerState {
      data: WeatherData | null;
      error: string | null;
      isPolling: boolean;
      isRetrying: boolean;
    }

    export function useWeather(
      fetchFn: () => Promise<WeatherData>,
      options?: { onImpactChange?: (impact: WeatherImpact) => void }
    ): WeatherPollerState
    ```

    **Constants (different from useMatchPoller):**
    - `POLL_INTERVAL = 600_000` (10 min, not 30s) (D-17-10)
    - `MAX_RETRIES = 3`
    - `RETRY_BASE = 1_000`

    **Hook internals (copy verbatim from useMatchPoller):**
    - `useRef<typeof state>(state)` for `stateRef` (to read latest state in doFetch without re-creating it)
    - `retryCountRef`, `intervalRef`, `retryTimerRef`, `isVisibleRef`, `fetchFnRef` — exact same pattern
    - `useEffect` to keep fetchFnRef.current in sync
    - `clearTimers` — clears both interval and retry timers
    - `doFetch` — the core fetch logic (see below for additions)
    - `handleVisibilityChange` — same visibility API pause/resume with immediate doFetch
    - Main `useEffect` — registers visibilitychange listener, starts initial poll and interval
    - Cleanup: removes listener, clears timers

    **Additions to doFetch (beyond useMatchPoller):**
    - On successful fetch, before setState: compare `result.impact` with `stateRef.current?.data?.impact` (using ref to avoid stale closure)
    - If impact changed and `onImpactChange` is provided, call `onImpactChange(result.impact)`
    - Update state with `data: result`, `error: null`, `isRetrying: false`

    **Error message:** `"Weather data unavailable — retrying..."` (per D-17-08)

    **isFetching lock:** Add `isFetchingRef = useRef(false)`. At the top of doFetch, `if (isFetchingRef.current) return` and set it to true, then reset in finally block. This prevents race conditions on visibility change rapid toggle (Pitfall 3 from RESEARCH).

    **Important differences from useMatchPoller:**
    - No `MatchState` import — use `WeatherData` from `@/types/weather`
    - `PollerState` → `WeatherPollerState` (same fields, different data type)
    - Accepts optional `options` parameter with `onImpactChange` callback
    - Does NOT write to weatherSlice directly — the hook returns data to the consumer (DashboardPage manages store writes)
